C++ lifetime and ownership

Reason from invariants: who owns each resource, how long every view remains valid, and what happens on every exit path.

Core: Days 1-3 C++20 Interview anchor
Mission fit. A senior systems answer should make ownership and failure paths obvious before discussing clever syntax. Use this guide while building the frame codec and move-only file-descriptor wrapper.
Two-week boundary. Core means safe value types, RAII, move-only handles, and non-owning views. Allocator internals, placement construction, intrusive ownership, and exotic `shared_ptr` patterns are stretch topics, not implied mastery.

The four-question model

  1. What is the resource?
    Object, allocation, file descriptor, lock, socket, thread?
  2. Who owns it?
    Exactly one type should state the release obligation.
  3. What is borrowed?
    Pointer, reference, `span`, or `string_view`; document validity.
  4. Which invariant survives?
    Normal return, exception, move, cancellation, and error.

Storage and object lifetime are related but not identical. "Stack" and "heap" describe storage strategies; neither says who owns a resource. The language defines when an object's lifetime begins and ends, and using storage as though an object were alive outside that interval can be undefined behavior. Use the public draft's object lifetime rules when exact semantics matter.

The standard descriptions of unique_ptr and span encode the essential contrast: an owner releases; a view never extends lifetime.

RAII: make illegal cleanup paths hard to express

Resource Acquisition Is Initialization binds acquisition to object construction and release to destruction. Prefer the Rule of Zero: compose types that already manage themselves. If a class directly owns a raw resource, define or delete the special members deliberately (the Rule of Five).

#include <unistd.h>
#include <utility>

class UniqueFd {
public:
  UniqueFd() noexcept = default;
  explicit UniqueFd(int fd) noexcept : fd_(fd) {}
  ~UniqueFd() { reset(); }

  UniqueFd(const UniqueFd&) = delete;
  UniqueFd& operator=(const UniqueFd&) = delete;

  UniqueFd(UniqueFd&& other) noexcept
      : fd_(std::exchange(other.fd_, -1)) {}

  UniqueFd& operator=(UniqueFd&& other) noexcept {
    if (this != &other) reset(std::exchange(other.fd_, -1));
    return *this;
  }

  [[nodiscard]] int get() const noexcept { return fd_; }
  [[nodiscard]] explicit operator bool() const noexcept { return fd_ >= 0; }
  [[nodiscard]] int release() noexcept { return std::exchange(fd_, -1); }

  void reset(int next = -1) noexcept {
    const int old = std::exchange(fd_, next);
    if (old >= 0) ::close(old); // destructor cannot report close failure
  }

private:
  int fd_ = -1;
};

The moved-from state remains valid and non-owning. Move operations are noexcept, allowing containers to relocate values safely. In production, operations requiring a reported close/flush error need an explicit method before destruction; a destructor must not throw. The C++ Core Guidelines' resource rules are a useful review checklist.

Resource invariant checklist

  • One owner has the release responsibility.
  • Invalid and moved-from states are explicit and harmless.
  • Copy is either value-correct or deleted; move transfers exactly once.
  • Construction failure leaks nothing; partial work rolls back.
  • Borrowed views cannot outlive their backing storage by contract or construction.
  • Destruction is non-throwing and safe on every exit path.

Error safety is an invariant, not an exception slogan

GuaranteeAfter failureTypical technique
No-throwOperation cannot fail through an exceptionDestructors, swaps, narrow moves
StrongObservable state is unchangedBuild temporary, validate, then commit/swap
BasicInvariants hold, but state may changeRAII guards and carefully ordered mutation
No guaranteeInvariants may be brokenA design smell in reusable systems code

Modern C++20 is most useful when it exposes contracts: values by default, std::span for buffer views, std::optional for optional values, std::variant for closed alternatives, and constrained templates where generic code is warranted. Do not add abstractions only to advertise a language feature.

Interview traps

TrapWhy it failsStronger answer
"A raw pointer owns the object."The syntax carries no ownership semantics.State ownership separately; use raw pointers/views primarily for borrows.
"std::move moves the object."It is a cast enabling move overload resolution; the selected operation does the transfer.Describe the type's move constructor and valid moved-from invariant.
"Use shared_ptr to be safe."Shared lifetime adds allocation, atomic control-block traffic, cycles, and unclear ownership.Prefer values or unique ownership; justify shared lifetime explicitly.
"A shared_ptr makes the object thread-safe."Control-block operations can be synchronized while the pointed-to object's data still races.Separate lifetime safety from state synchronization.
Return a string_view into a temporary/local string.The view outlives the characters and dangles.Return an owning string or tie the view to longer-lived storage.
"The destructor handles every error."Destructors cannot reliably report failures such as final flush/close errors.Provide an explicit commit/close when the result matters; retain RAII fallback cleanup.
delete this as normal lifetime management.Callers and aliases can keep using a destroyed object; ownership becomes opaque.Put destruction under an explicit owner.

Stretch shelf

After the core project is safe: allocator propagation and alignment, custom deleters, placement construction, polymorphic memory resources, shared_ptr aliasing, intrusive reference counting, and cache-aware layout. These are role-dependent depth topics, not prerequisites for the capstone.
