Debugging and performance

Turn symptoms into evidence, evidence into a falsifiable hypothesis, and each change into a measured comparison.

Core: Day 10 Reuse: Day 14 Linux / WSL + macOS
Mission fit. Technical rounds often test how you investigate, not just whether you remember a command. Practice one crash, one race, one hang, one syscall failure, and one throughput bottleneck in the capstone.
Two-week boundary. A repeatable evidence-first workflow and the core tools are required. eBPF, production fleet profiling, hardware-counter expertise, and advanced allocator/NUMA tuning are stretch.

The evidence loop

  1. Reproduce.
    Smallest reliable input, build, environment, and timing.
  2. Observe.
    Logs, return values, backtraces, syscalls, counters, capture.
  3. Hypothesize.
    One causal claim with a prediction that can be wrong.
  4. Instrument.
    Use the narrowest tool that distinguishes alternatives.
  5. Change.
    One controlled fix; preserve a regression test.
  6. Compare.
    Same workload, uncertainty shown, side effects checked.
"It is slow" is a symptom. "At 500 connections, p99 rises with runnable threads while CPU is saturated in frame copies" is evidence you can design against.

ASan and TSan are normally separate configurations. Always preserve the exact compiler, flags, commit, workload, and environment with a result. Official CMake Presets make this repeatable across VS Code and the terminal.

Fast diagnostic playbooks

Crash

  1. Capture exit status, signal, stderr, exact input, and build identity.
  2. Reproduce under ASan/UBSan. If only optimized/production-like, retain a core dump.
  3. In GDB/LLDB, inspect all thread backtraces, faulting frame, arguments, object invariants, and ownership history.
  4. Reduce to the smallest failing case; write the regression before or with the fix.

Hang or deadlock

  1. Define no-progress precisely: no accept, no completed request, queue unchanged, or thread blocked?
  2. Capture all thread stacks more than once. Map each wait to its mutex, condition, FD, or child.
  3. Use syscall tracing to distinguish user-space lock waits from blocking I/O.
  4. Check lock order, missed predicate transitions, lost wakeup assumptions, queue bounds, and shutdown paths.

Network failure

  1. Record endpoint, socket state, syscall result and errno, deadline, and byte counts.
  2. Correlate application logs with strace and a packet capture using timestamps/request IDs.
  3. Separate name resolution, route, handshake, TLS, framing, service, and application timeout failures.

Performance requires a question and a budget

MetricMeaningCommon mistake
LatencyTime per operation; report a distribution such as p50/p95/p99/maxReporting only the mean
ThroughputCompleted useful work per unit timeCounting attempts rather than successful work
UtilizationFraction of a resource's available capacity usedAssuming high utilization identifies the cause
SaturationNo spare capacity; queued work or contention growsIgnoring queue delay while CPU looks acceptable
ErrorsFailures, timeouts, rejections, retries, corruptionsCalling shedding failures without separating policy

Measure CPU profile, syscalls, allocations, context switches, page faults, cache-miss indicators, queue depths, and bytes copied only when they test a bottleneck hypothesis. The C++ Core Guidelines' performance section reinforces: do not optimize without data.

Benchmark hygiene

  • Workload is representative and input/output cannot be optimized away.
  • Warmup, run duration, repetitions, and summary statistics are recorded.
  • Hardware/VM/WSL version, compiler, flags, and background load are controlled or disclosed.
  • Correctness, error rate, and resource use are measured beside speed.
  • Before/after uses the same harness and includes uncertainty/noise.
  • Optimization target is tied to a user or system budget.

Interview traps

TrapWhy it failsStronger answer
Optimize the hottest-looking code firstIntuition routinely misidentifies end-to-end bottlenecks.State budget, reproduce load, profile, then change.
"The sanitizer is clean, so the code is safe."Dynamic tools see executed paths and have coverage/implementation limits.Combine invariants, tests, review, multiple tools, and fault injection.
Run ASan and TSan as one magic buildThey have incompatible or impractical instrumentation combinations.Use explicit independent presets and the same scenario suite.
Use average latencyQueues and rare stalls hide in the tail.Show distribution, percentiles, max, load, and sample count.
Assume debugger behavior equals normal timingBreakpoints and instrumentation alter schedules and deadlines.Capture cores/traces and reproduce with minimally invasive tools.
Read a hardware counter as a conclusionCounters require normalization and causal context.Connect counter changes to profile, code path, and controlled experiment.
Microbenchmark an isolated function and claim service speedupEnd-to-end queues, I/O, contention, and copies may dominate.Validate the change in both micro and representative service workloads.

Glossary

TermCourse meaning

LatencyTime taken per operation; always qualify the distribution and load.
ThroughputUseful work completed per unit time.
SaturationA resource has no spare capacity and waiting/queueing grows.
