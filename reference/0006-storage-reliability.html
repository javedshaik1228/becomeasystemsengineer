Storage and reliability

A write is not a durability promise. Name the layer, ordering, acknowledgement point, and failure model before saying data is safe.

Core: Day 11 Stretch: NAS orientation Failure-first
Mission fit. Add a tiny write-ahead log to NetForge, inject truncation and restart failures, and explain exactly what an acknowledged mutation survives.
Two-week boundary. Core is local-file durability mechanics, a small WAL, recovery invariants, and failure injection. NFS/SMB internals, distributed consensus, RAID implementation, iSCSI, NVMe, and high-availability storage are specialist orientation only.

Place the bytes in the stack

  1. Application.
    Object state, serialization, user-space buffers.
  2. System call / VFS.
    Kernel interface and filesystem abstraction.
  3. Page cache / filesystem.
    Dirty pages, metadata, journaling/COW policy.
  4. Block stack / device.
    Queues, controller cache, media, flush semantics.
  5. Remote path.
    Client/server caches, protocol acknowledgement, network.

The Linux kernel's VFS documentation describes the abstraction between applications and concrete filesystems. A successful C++ stream flush, write(), or close() can leave bytes only in volatile layers. Define durability relative to failures such as process crash, kernel panic, power loss, or device loss.

Durable replace pattern

  1. Create a temporary file in the same directory, with safe flags and permissions.
  2. Write all bytes; handle short writes, EINTR, disk full, quota, and I/O errors.
  3. Call fsync(temp_fd) (or the platform/filesystem-appropriate equivalent) and check it.
  4. Atomically rename(temp, target) in the same filesystem.
  5. Open and fsync the containing directory so the directory-entry update is persisted.

This is a pattern, not a universal proof: filesystem, mount options, device cache, hardware behavior, and the exact failure model matter. Linux fsync(2) notes that syncing a file does not necessarily persist its directory entry; rename(2) defines namespace replacement semantics but not, by itself, every durability guarantee.

// Conceptual recovery loop.
while (auto header = read_header(log)) {
  if (!valid_version(*header) || header->length > kMaxRecord) break;
  auto payload = read_exact(log, header->length);
  if (!payload || !checksum_ok(*header, *payload)) break; // torn tail
  if (!already_applied(header->operation_id)) {
    apply_idempotently(*header, *payload);
  }
}

// state and report the policy instead of silently skipping arbitrary bytes.

For the two-week project, keep the model small: append-only length-delimited records, checksum, operation ID, explicit sync mode, deterministic replay, and injected truncation. Write down whether acknowledgement means process-crash durability, host power-loss durability, or something weaker.

Interview traps

TrapWhy it failsStronger answer
"close() means durable."It releases a descriptor; dirty data may remain in volatile caches and errors may surface elsewhere.Name and check the required durability barrier before acknowledgement.
"fsync(file) persists the filename."The directory entry is separate metadata.For durable creation/replacement, sync the containing directory as required.
"Atomic rename makes the update power-safe."Namespace atomicity and durable persistence are different properties.Order file sync, rename, and directory sync under a declared failure model.
Assume write() transfers everythingShort writes and errors are allowed.Loop with checked progress; handle full disk and unknown outcomes.
"RAID is a backup."It does not protect against deletion, corruption, compromise, or all correlated failures.Separate availability, redundancy, snapshots, and independent tested backups.
"Replication means strong consistency."A copy count does not specify acknowledgement or read ordering.Name the consistency model and quorum/failover behavior.
Assume NFS/SMB behaves exactly like a local filesystemNetwork, cache, state, lock, and recovery semantics add failure modes.Ask which protocol/version/mount policy and reason from its contract.
