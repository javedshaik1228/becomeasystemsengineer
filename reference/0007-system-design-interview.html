Systems design interview guide

Design a service as a set of explicit contracts: requirements, invariants, capacity, overload, failure, observability, and justified tradeoffs.

Core: Days 13-14 45-minute map NetForge capstone
Mission fit. Use one coherent design - the NetForge framed TCP key/value service - to demonstrate modern C++, threads, sockets, protocol hardening, durability, debugging, and operational judgment.
Two-week boundary. The core goal is a defensible single-node service and clear scale/failure extensions. NAS internals, routing control planes, cryptographic protocol design, consensus, and multi-region storage remain specialist stretch areas.

Capacity math you can defend

active_memory ~= connections * per_connection_bytes
              + queued_jobs * average_job_bytes
              + key_count * average_entry_bytes
              + fixed_process_overhead

required_service_rate > peak_arrival_rate / target_utilization
queue_wait ~= queue_depth / effective_completion_rate  // rough diagnostic

daily_log_growth ~= writes_per_second * average_record_bytes * 86,400
fd_budget > listeners + active_connections + WAL/files + diagnostics + reserve

Example assumptions are useful only when labeled: 20,000 connections at 32 KiB of total connection state is about 625 MiB before allocator/other overhead; a 10,000-item queue at 8 KiB average is about 78 MiB. Bounds are architecture, not just validation.

Concurrency architecture choices

ModelBest fitRiskDefensible NetForge use
Thread per connectionSmall connection count, blocking simplicityMemory/scheduling growth and slow-client occupancyFirst correctness baseline and comparison point
Fixed worker poolBound CPU/blocking work with a queueQueue overload and head-of-line blockingBounded request jobs; explicit rejection/drain policy
Reactor/event loopMany mostly idle connections, nonblocking I/OState-machine complexity and accidental blockingPer-connection input/output buffers; offload bounded work
HybridHigh connection count plus CPU/storage workCross-thread ownership and overload couplingOne/few I/O loops + bounded workers; ownership transfer documented

A clean data path is: accept/admit connection → incrementally parse bounded frame → authorize/validate → enqueue bounded work or execute → mutate/read store → append/sync WAL if promised → enqueue bounded response → write partial bytes → record outcome. Mark every queue and owner on the diagram.

Overload is normal operation at the boundary

MechanismPolicy questionFailure avoided
Connection capReject immediately, backlog briefly, or prioritize clients?FD/memory exhaustion
Frame and buffer capProtocol error, close, or drain malicious/oversized input?Memory and parsing abuse
Bounded work queueBlock producer, reject, shed low priority, or degrade?Unbounded latency/memory
DeadlineEnd-to-end or per-hop? What cleanup/cancellation occurs?Zombie work and resource retention
Retry policyIs operation idempotent? Backoff, jitter, budget, max attempts?Retry amplification and synchronized storms
Output watermarkPause reads, drop response, or close a slow reader?Per-client memory takeover

Google SRE's overload guidance and the AWS Builders' Library on timeouts, retries, backoff, and jitter are high-trust operational anchors. Backpressure must propagate or terminate at a deliberate shedding boundary.

Observe outcomes and saturation

The Google SRE chapter on monitoring distributed systems motivates four useful signal families. For NetForge:

SignalMinimum useful measurements
TrafficAccepted/active connections, requests and bytes by operation
ErrorsParse failures, rejected work, timeouts, resets, WAL/sync failures, recovery corruption
DurationEnd-to-end and queue/service latency distributions by operation/outcome
SaturationWorker utilization, queue depth/oldest age, output-buffer bytes, FDs, CPU, memory, disk latency/capacity

Use structured logs with request/connection/operation IDs, bounded-cardinality dimensions, and enough context to join application, syscall, and packet evidence. Never log secrets or unbounded attacker-controlled payloads.

Interview traps

TrapWhy it failsStronger answer
Draw architecture before clarifying requirementsThe design cannot be evaluated without load and guarantees.Prioritize requirements, assumptions, and non-goals first.
Leave a queue unboundedIt converts overload into memory growth and extreme latency.State capacity, admission, shedding, and metric for every queue.
Retry every timeoutUnknown outcomes and retry amplification can worsen failure.Require idempotency/request IDs, budget, backoff, jitter, and deadline.
Say "strong consistency" without scopeThe term does not define operations, order, replicas, or failures.Name the exact observable guarantee and acknowledgement point.
Choose epoll as the scaling answerReadiness does not solve CPU, storage, contention, slow peers, or queueing.Use capacity evidence and an end-to-end bottleneck model.
Report only average latencyTail delay reveals queues, contention, retries, and outliers.Specify percentile, load, errors, and saturation together.
Name technologies without tradeoffsIt hides the decision criteria and operational cost.Record choice, rejected option, evidence, and revisit trigger.
Ignore shutdownThreads, descriptors, accepted work, and durability promises become unsafe.Walk stop-admit, wake, drain/deadline, join, sync, release.

Glossary

TermCourse meaning
InvariantA property that must remain true across all allowed operations and failure paths.
BackpressureLimiting or admitting work so demand cannot grow without bound.
Load sheddingDeliberately rejecting or degrading work to preserve bounded service for admitted work.
IdempotentApplying an operation more than once has the same allowed effect as applying it once.
DurabilityAcknowledged data survives the explicitly stated failure model.
LatencyTime taken per operation; always qualify the distribution and load.
ThroughputUseful work completed per unit time.
SaturationA resource has no spare capacity and waiting/queueing grows.
