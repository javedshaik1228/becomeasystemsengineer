Concurrency and the memory model

Make shared state boring: assign ownership, bound work, synchronize invariants, and give every thread a stopping story.

Core: Days 4-5 C++20 TSan practiced
Mission fit. The interview target is not memorizing every memory order. It is designing a race-free bounded queue and thread pool, defending the synchronization, and diagnosing a deliberately injected race.
Two-week boundary. Mutexes, condition variables, structured thread lifetime, cancellation, and ordinary atomics are core. Lock-free algorithms, memory reclamation, NUMA tuning, and weak-memory proofs are stretch.

Start with state, not threads

  1. Partition state.
    Prefer immutable, thread-confined, or message-passed data.
  2. Name each invariant.
    Which fields must change atomically as a group?
  3. Choose synchronization.
    Mutex/CV first; atomics for narrow independent state.
  4. Specify shutdown.
    Who requests stop, wakes waiters, joins, and rejects new work?

Threads in one process share address space; each thread also has its own execution state and stack. The C++ model defines a data race as conflicting potentially concurrent actions, at least one a write, without the required atomicity or ordering. A data race is undefined behavior, not merely a stale-read risk. See the draft's multithreaded execution rules.

Happens-before is the visibility proof

Producer actionSynchronizing edgeConsumer consequence
Write protected state, then unlock mutexUnlock synchronizes with a later successful lock of that mutexLater lock holder sees preceding protected writes
Modify predicate under mutex, then notifyWait atomically releases/reacquires mutex; predicate is checked while lockedWaiter observes a state satisfying the predicate, not a remembered notification
Initialize data, then release-store flagAcquire-load observes that release sequenceWrites before release become visible after acquire
Thread function returnsSuccessful join()Joining thread observes completed thread effects

The standard defines the complete atomic ordering relations. In an interview, say which edge carries visibility. "There is a mutex" is not enough if some accesses bypass it.

The bounded queue pattern

// not_empty_ is std::condition_variable_any for the stop-token overload.
std::optional<Job> pop(std::stop_token stop) {
  std::unique_lock lock(mu_);
  not_empty_.wait(lock, stop, [&] {
    return closed_ || !queue_.empty();
  });

  if (queue_.empty()) return std::nullopt; // closed or stopped
  Job job = std::move(queue_.front());
  queue_.pop_front();
  lock.unlock();
  not_full_.notify_one();
  return job;
}

bool push(Job job) {
  std::unique_lock lock(mu_);
  not_full_.wait(lock, [&] {
    return closed_ || queue_.size() < capacity_;
  });
  if (closed_) return false;
  queue_.push_back(std::move(job));
  lock.unlock();
  not_empty_.notify_one();
  return true;
}

The predicate owns correctness: condition-variable notifications are hints and wakeups can be spurious. The queue's invariant might be size <= capacity, and all fields participating in that invariant are read and written under one mutex. Move expensive work out of the critical section. C++20's std::jthread joins on destruction and carries a cooperative stop mechanism, but blocking APIs still need a wakeup path.

Pick the primitive that protects the invariant

PrimitiveGood fitWarning
MutexSeveral fields form one invariant; operations are compoundKeep lock ownership and scope visible
Condition variableWait until protected state satisfies a predicateAlways wait with a predicate in a loop
SemaphoreCount available permits/resourcesDoes not itself protect related container state
AtomicCounter, flag, pointer, or carefully proven state machineAtomic fields do not make a compound invariant atomic
Message passingTransfer ownership and isolate mutable stateThe channel still needs bounds, shutdown, and failure policy

Use the simplest order that is correct. Sequential consistency is a defensible default for small coordination atomics. Move to acquire/release or relaxed only with a documented proof and measurement. memory_order_relaxed provides atomicity for that object, not cross-object ordering.

Deadlock and liveness review

  • Establish a global lock order; acquire multiple mutexes with std::scoped_lock where suitable.
  • Never call unknown callbacks, blocking I/O, logging pipelines, or user code while holding a state mutex.
  • Bound waits with a timeout only when the timeout has defined semantics; timing out is not automatic recovery.
  • Distinguish deadlock, livelock, starvation, and slow progress using thread backtraces and counters.
  • Run a dedicated ThreadSanitizer build. Keep it separate from ASan, and exercise real concurrent paths.

Interview traps

Use volatile for thread synchronizationIt does not provide inter-thread atomicity or happens-before in C++.Use mutexes or atomics with an explicit ordering argument.

Stretch shelf

Acquire/release publication patterns, lock-free queues, ABA and reclamation, false sharing, cache-line padding, NUMA locality, work stealing, and formal progress guarantees. Learn to recognize these terms; do not claim a lock-free implementation without a memory-order proof and adversarial tests.

Glossary

TermCourse meaning
InvariantA property that must remain true across all allowed operations and failure paths.
Data raceConflicting non-atomic accesses in different threads without a happens-before relationship, with at least one write.
Happens-beforeThe ordering relation used to prove that one operation's effects are visible to another.
Critical sectionCode that accesses state protected by a synchronization primitive.
BackpressureLimiting or admitting work so demand cannot grow without bound.
