Recall before reading

Why would one synchronous send invalidate your matching-latency claim? What happens when the egress queue is full?

Mental model

The engine emits immutable fills into bounded egress. A separate worker serializes and publishes them. Redis RESP commands are arrays of bulk strings; PUBLISH returns an integer receiver count. Kafka acknowledgements and retries belong to the adapter policy, not the matching core; review the official Kafka guarantees.

Build

  1. Implement byte-deterministic serialize_fill_v1.
  2. Add a FillSink port and fake slow/failing sink.
  3. Put a bounded egress queue between engine and sinks.
  4. Implement Redis RESP2 PUBLISH with write-all semantics and parsed integer reply.
  5. Add optional librdkafka or sidecar publishing behind the same port.

Break it deliberately

Make the sink sleep, fail every third publish, accept partial socket writes, and fill the egress queue. The book must remain correct and failures observable.

Hint 1 · test without brokers
