Recall before reading

Where does Order ownership transfer? Why must a bounded service expose “full” rather than allocate forever?

Mental model

A producer claims a position, writes the value, then releases the slot sequence. A consumer acquires that sequence before reading the value. Release/acquire makes preceding writes visible across threads; review the C++ atomic ordering rules.

Build

  1. First make a sequential bounded ring pass empty, full, and wraparound tests.
  2. Round capacity to a documented power of two.
  3. Add one atomic sequence number per cell and atomic enqueue/dequeue positions.
  4. Implement try_push and try_pop without hidden blocking.
  5. Run repeated multi-producer/single-consumer tests and ThreadSanitizer.

Break it deliberately

Use a capacity of two, wrap thousands of times, enqueue move-only values, and force producers to observe full. Explain why lock-free does not mean wait-free.

Hint 1 · sequence states

A free cell’s sequence matches its enqueue position; a published cell’s sequence is one greater. After consumption, advance it by capacity.

Hint 2 · memory ordering

Start with sequentially consistent atomics for correctness, then justify each weaker order with a happens-before argument.

Stuck protocol. At 20 minutes, get a sequential ring green. At 40, draw one slot through free → claimed → published → consumed. At 60, inspect only the reference cell-sequence transitions.
