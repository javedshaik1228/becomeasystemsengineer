Recall before reading

Why is double dangerous for equality and ordering? Which fields make an order uniquely addressable? Say the difference between a command and an emitted event.

Mental model

An Order is an input command. A Fill is a fact created by matching. Prices are signed integer ticks; quantities and IDs are unsigned counts. Invalid values must be rejected before they enter any book. Review the C++ standard’s fundamental types.

Build

  1. Read the scaffold in work/include/matching/order.hpp.
  2. Implement valid_order: nonzero ID, nonempty symbol, positive price, nonzero quantity, nonzero sequence.
  3. Add two boundary cases of your own. Do not add an order book yet.

Break it deliberately

Try zero values, a negative price, an empty symbol, and the largest representable quantity. Explain whether each failure is representational or business policy.

Hint 1 · keep policy explicit

The starter’s rule is intentionally small. Do not invent exchange-specific price bands yet.

Hint 2 · failing first

The scaffold returns false for everything. Make one valid case green before adding invalid cases.

Stuck protocol. At 20 minutes, write the invariant in plain English. At 40, inspect only the type declarations. At 60, ask the agent with the smallest failing value; do not open the solution.
