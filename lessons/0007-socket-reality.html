The model: lifecycle plus byte-stream protocol

  1. Resolve addresses
  2. Create socket
  3. Bind + listen
  4. Accept client
  5. Read bounded frame
  6. Execute + reply

getaddrinfo produces address candidates. The listener is long-lived; each successful accept creates a new connected descriptor with one owner. Day 7's gate requires one request; the reference can serve sequential framed requests on that connection so Day 8 can measure persistent sessions. Each active connection still occupies one blocking worker with finite I/O timeouts. This is deliberately simpler than a reactor and exposes slow-client trade-offs for Day 8.

Read socket(7), the relevant call pages it links, and the course's POSIX I/O and sockets sheet.
