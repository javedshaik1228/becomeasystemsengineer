RAII · move semantics · failure paths

Never leak a descriptor

Make a socket impossible to copy, safe to move, and boring to release—even when construction, parsing, or a later syscall fails.

150 minutesGate: day02Output: UniqueFd

Recall before reading

Without notes: list every exit path from a function that manually calls close. Which compiler-generated special members become wrong when an integer represents unique ownership? Why should a destructor avoid throwing?

The model: ownership is a type invariant

A descriptor is a small integer naming a process resource. The integer is copyable; the right to close it must not be. Wrap that right in a move-only object whose destructor releases once, whose moved-from state is invalid, and whose release/reset operations make transfer explicit. This is RAII applied beyond heap memory.

Raw handles can be non-owning observations. The owner is the object whose lifetime performs the matching release.

Read C++ Core Guidelines R.1, then the short ownership reference. On Linux, close(2) warns why blindly retrying close after EINTR can close a reused descriptor.
