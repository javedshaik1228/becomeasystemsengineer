Threads with lifetimes

Turn the queue into a joining worker pool with an explicit drain policy. A thread is a live owner relationship, not a fire-and-forget callback.

180 minutesGate: day05Output: ThreadPool

Recall before reading

What happens if a joinable std::thread dies? Which objects may a worker access, and therefore which must outlive it? Does requesting stop interrupt a blocking syscall? Which invariant do your metrics protect?

The model: accept, close, drain, join

  1. Construct queue
  2. Start joining workers
  3. Accept bounded tasks
  4. Close submissions
  5. Drain existing work
  6. Join every worker

std::jthread joins on destruction and carries cooperative stop machinery, but cancellation remains a protocol. NetForge's pool closes its queue, lets workers consume until closed-and-empty, then joins. No detached worker may observe destroyed store, WAL, or server state. Relaxed atomic counters are appropriate only because they report independent totals and do not publish other memory.

Use the standard's jthread clauses and atomic ordering clauses, plus the printable memory-model sheet.
